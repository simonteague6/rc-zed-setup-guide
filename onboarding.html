Your First Pull Request

This page walks you from a fresh laptop to your own profile card merged into the team's profile-card repo. No experience needed. Do one step at a time, check it off, and take breaks whenever you want — your checkmarks are saved in this browser, so you can close the tab and come back later.

🖥️
First: pick your operating system Use the macOS / Windows switch in the top bar. The page will show only the commands for your machine.

Fill these in once — the commands below will personalize themselves

STAGE 1

Install your tools

You only do this stage once, ever. It gets you a GitHub account, git (the tool that tracks changes to code), and the Zed editor.

This is the short version. The club's full Setup Guide covers the same steps with screenshots, plus optional shell niceties like Oh My Zsh — if anything here feels too fast, use that page for this stage and come back.

Create a GitHub account

GitHub is the website where the club stores its code. Go to github.com/signup and make a free account.

  • Use an email you'll still have after graduation (a personal one, not only your UTSA email — you can add both later).
  • Pick a username you'd be OK putting on a resume. You'll be typing it a lot.

Then scroll up and type your username, name, and email into the boxes at the top of this page.

Open Terminal and install the command line tools macOS

Press Cmd+Space, type Terminal, and hit Enter. That black window is where you'll paste commands.

Now paste this and press Enter (click a command's Copy button, then Cmd+V in Terminal):

xcode-select --install

Click Install in the popup and wait for it to finish. This gives you git and other developer tools macOS doesn't ship with. If it says the tools are already installed, that's fine — check this step off.

Install Zed macOS

Zed is the code editor the club uses. Go to zed.dev/download, download the .dmg, open it, and drag Zed into Applications.

💡 Already use Homebrew? Install it that way instead 
brew install --cask zed

The full Setup Guide recommends this path because brew upgrade keeps Zed updated. If you don't know what Homebrew is, the direct download above is perfectly fine.

Open Zed and turn on its terminal command macOS

Open Zed from Applications. Then press Cmd+Shift+P, type install cli, and press Enter. This lets you type zed in the terminal to open projects — you'll need it in Stage 3.

Zed also has a terminal built in: press Ctrl+` (the backtick key, left of 1). From now on, any command on this page can be pasted there.

Check virtualization is on Windows

On Windows you'll run your developer tools inside Linux, using something called WSL. That needs one hardware setting turned on. It's safe and won't void any warranty.

Open Task Manager (Ctrl+Shift+Esc) → Performance tab → CPU. Look for "Virtualization: Enabled" near the bottom right.

If it says Disabled, it needs to be flipped on in BIOS — the Setup Guide's "Before You Start" section has the BIOS key for every laptop brand. Also make sure Windows is up to date (Settings → Windows Update).

Install WSL Windows

Press the Start key, type powershell, and click Run as Administrator on the right. Then paste this and press Enter:

wsl --install

When it finishes, restart your computer. (Needs Windows 10 version 2004+ or Windows 11.)

Create your Ubuntu username and password Windows

After the restart, Ubuntu opens on its own (or find Ubuntu in the Start menu). It asks you to create a username and password. Pick anything, and remember the password — you'll type it whenever a command starts with sudo.

⌨️
The password won't show as you type No dots, no stars, nothing. The screen is not frozen — this is normal terminal behavior. Just type it and press Enter.
Update Ubuntu and install git Windows

In the Ubuntu window, run these two commands, one at a time. The first can take a few minutes — that's normal.

sudo apt update && sudo apt upgrade -y
sudo apt install git zsh -y

(zsh is a nicer shell the Setup Guide sets up fully — optional for today, but installing it now saves time later.)

Install Zed — the Windows app Windows

Zed is the code editor the club uses. Go to zed.dev/download, download the Windows installer, and run it.

⚠️
This is the Windows app Don't try to install Zed inside Ubuntu. You install it on Windows once, then connect it to Ubuntu in the next step.
Connect Zed to Ubuntu (WSL Remote) Windows

Open Zed. When opening a project, choose the WSL Remote option in the project picker — that connects Zed to your Ubuntu files. The Setup Guide has a screenshot of exactly where it is.

🛑
Two rules that prevent 90% of Windows problems 1. Every time you open a terminal, type wsl first, then cd ~. Otherwise you're in PowerShell and the commands on this page won't work.
2. Keep your projects in Ubuntu's home folder (~), never in paths that start with /mnt/c/. Always open projects in Zed through the WSL Remote.

Zed also has a terminal built in: press Ctrl+` (the backtick key, left of 1). When you're connected via WSL Remote, that terminal is already inside Ubuntu — no wsl needed there.

Tell git who you are

Every change you save with git gets stamped with a name and email. Set them once and git remembers forever. Run these in your terminal (inside Ubuntu — type wsl first if you opened Windows Terminal):

git config --global user.name "YOUR NAME"
git config --global user.email "you@example.com"

Use the same email as your GitHub account. If you filled in the boxes at the top of this page, these commands already have your info in them.

STAGE 2

Set up your GitHub SSH key

An SSH key is like a house key for GitHub: your computer proves it's you without typing a password every time. You make a key pair — a private half that never leaves your laptop, and a public half you give to GitHub.

Make your key

In your terminal (inside Ubuntu), run:

ssh-keygen -t ed25519 -C "you@example.com"

It asks three questions. Press Enter for all three — that accepts the normal file location and skips the extra passphrase (fine for a laptop only you use). You'll see a box of random characters called "randomart"; that means it worked.

Copy the public half of your key

This command puts your public key on the clipboard (it's safe to share — the private half stays in place):

pbcopy < ~/.ssh/id_ed25519.pub
cat ~/.ssh/id_ed25519.pub | clip.exe

If clip.exe complains, just run cat ~/.ssh/id_ed25519.pub, then select the whole output line with your mouse and copy it.

Nothing appears to happen when you run it — that's normal. The key is on your clipboard now.

Give the public key to GitHub
  1. Open github.com/settings/keys (sign in if asked).
  2. Click New SSH key.
  3. Title: anything, like my laptop. Key type: leave as Authentication Key.
  4. Click in the Key box and paste (Cmd+VCtrl+V). It should start with ssh-ed25519.
  5. Click Add SSH key.
Test it 
ssh -T git@github.com

The first time, it asks if you're sure you want to connect — type yes and press Enter. Then you should see:

Hi your-username! You've successfully authenticated, but GitHub
does not provide shell access.
✅
"does not provide shell access" is not an error That whole message means it worked. If you instead see Permission denied (publickey), open the troubleshooting section at the bottom of this page.
STAGE 3

Get the team repo onto your laptop

"Cloning" downloads a copy of the team's repository — including its whole history — onto your machine. You'll clone profile-card, a small practice repo where every member adds their own profile page.

Accept your invite to the repo

Simon adds each member as a collaborator on simonteague6/profile-card. GitHub emails you an invitation — check your inbox (and spam), or look at github.com/notifications, and click Accept.

No invite? Message Simon with your GitHub username. You can keep going — you'll need the invite accepted before Stage 7 (push).

Clone the repo

In your terminal (inside Ubuntu — remember: type wsl then cd ~ if you opened Windows Terminal), run these one at a time:

mkdir -p ~/projects
cd ~/projects && git clone git@github.com:simonteague6/profile-card.git

The first command makes a projects folder in your home directory (it's fine if it already exists). The second downloads the repo into it.

⚠️
Windows: always clone in the terminal Don't use Zed's command palette to clone — it opens a Windows file picker that puts the project on the wrong filesystem. The terminal commands above put it in the right place.
Open the project in Zed 
cd ~/projects/profile-card && zed .

You should see index.html, style.css, and README.md in Zed's file tree on the left. That's the existing example card.

If zed isn't found inside Ubuntu, open Zed on Windows instead, use Open Folder through the WSL Remote, and pick ~/projects/profile-card.

If it says zed: command not found, open Zed, press Cmd+Shift+P, run install cli, and try again.

🔍 Curious? See the repo's history

This repo was built commit by commit, with two branches that split off and merged back. Run this to see the shape of it:

git log --oneline --graph --all

You're about to add your own branch to that picture. (Press q to exit the log view if it takes over the terminal.)

STAGE 4

Make your own branch

main is the team's shared copy — you never work directly on it. A branch is your own private workspace: you can experiment freely, and nothing touches main until the team approves your pull request.

Check where you are

From now on, run everything in Zed's built-in terminal (Ctrl+`) — it's already inside the project folder.

git status

You want to see On branch main and nothing to commit, working tree clean. That means you're starting from a clean copy of the team's code.

Create your branch 
git switch -c feature/add-your-username-card

switch -c means "create this branch and move me onto it." The feature/ prefix is the naming style this repo already uses. Zed's top bar now shows your branch name instead of main.

STAGE 5

Build your profile card

Your first contribution: one HTML file, your own page, in a cards folder. Everyone gets their own file, named after their GitHub username, so nobody steps on anyone else's work.

Make the cards folder 
mkdir -p cards

If someone's card was merged before yours, the folder already exists — -p makes that a non-issue either way.

Create your file 
touch cards/your-username.html

Now click the new file in Zed's file tree (inside the cards folder) to open it. It's empty — that's next.

Paste the template and make it yours

Copy this into your empty file. It follows the same structure as the repo's existing index.html and reuses the team's stylesheet:

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <link rel="stylesheet" href="../style.css">
</head>
<body>
  <div class="card">
    <h1>Hello, I'm YOUR NAME</h1>
    <p class="bio">One sentence about who you are.</p>
    <p class="goals">One sentence about what you want to learn this semester.</p>
  </div>
</body>
</html>

Then edit the two sentences so they're actually about you. Save with Cmd+SCtrl+S.

Look at it in your browser 
open cards/your-username.html
explorer.exe cards

That opens the folder in Windows File Explorer — double-click your file to open it in your browser.

You should see a small white card with your name on a gray background. If it looks unstyled (plain black text on white), check the stylesheet line says ../style.css — with the two dots.

STAGE 6

Commit your work

A commit is a saved snapshot with a message — like a checkpoint in a game. Right now your new file only exists on your laptop and git isn't tracking it yet. Committing happens in two moves: stage the file (put it in the box), then commit (seal the box with a label).

See what git sees 
git status

Your file shows up in red under "Untracked files." Red just means "git isn't tracking this yet" — nothing is wrong.

Stage your file 
git add cards/your-username.html

Run git status again if you're curious — the file turned green, meaning it's staged and ready to be committed.

Commit it 
git commit -m "feat: add your-username's profile card"

The -m part is the message — a short note about what this snapshot contains. The feat: prefix means "this adds a feature"; it's the style this repo's history already uses.

STAGE 7

Push your branch to GitHub

Your commit exists only on your laptop. Pushing uploads your branch to GitHub so the team can see it.

Push 
git push -u origin feature/add-your-username-card

origin is git's name for the GitHub copy of the repo, and -u links your local branch to it (you only need -u this first time).

In the output, GitHub prints a link that says "Create a pull request" — that's your cue for the next stage. If the push is rejected instead, open the troubleshooting section at the bottom.

STAGE 8

Open your pull request

A pull request (PR) is how you ask the team to pull your branch into main. It's a conversation, not a submission for a grade — someone looks at your change, maybe suggests tweaks, then merges it.

Start the pull request

Go to github.com/simonteague6/profile-card. There's a yellow banner with your branch name and a "Compare & pull request" button — click it.

No banner? Click the Pull requests tab → New pull request. Set base to main and compare to your branch, then Create pull request.

Write the title and description

Title: Add YOUR NAME's profile card

Description: a sentence or two is plenty. What you added, and anything you're unsure about. For example: "Adds my profile card in cards/. First PR — let me know if I got the file location right!"

Click "Create pull request"

That's it — your work is officially up for review. Simon gets a notification automatically. If nothing happens in a day or two, it's completely normal to ping him: "Hey, my PR is up, can you take a look?"

STAGE 9

Respond to review — then celebrate

A reviewer will read your change and either approve it or leave comments asking for tweaks. Everyone — including people who've coded for decades — gets change requests. It means someone read your work carefully.

Read the feedback

GitHub emails you when someone comments. On your PR page, the Files changed tab shows comments pinned to specific lines. If a comment is unclear, reply and ask — that's what the thread is for.

If the review is just an approval with no changes requested, check this off and skip to the last step. 🎉

Make the requested changes

Back in Zed, on the same branch (check the top bar — no new branch needed), edit your file, save, then:

git add cards/your-username.html
git commit -m "fix: address review feedback"
git push

No -u or branch name needed this time — git remembers from Stage 7. The PR updates by itself the moment you push.

After the merge: sync up

When your PR is merged, GitHub shows a purple Merged badge. Your card is now part of the team's main. Bring your laptop's copy up to date:

git switch main && git pull

Your branch did its job and can be deleted (this only removes the local copy; the history is safe in main):

git branch -d feature/add-your-username-card

You just did the full loop — branch, build, commit, push, PR, review, merge. That is exactly how real software teams work, and now you've done it.

HELP

When things go wrong

Everything in here has happened to every developer ever. Find your error, follow the fix, and if you're still stuck after 20 minutes, ask in the club chat or grab Simon at a meeting — asking early is the pro move, not the embarrassing one.

🔑 "Permission denied (publickey)" when testing SSH, cloning, or pushing

GitHub doesn't recognize your key. Check these in order:

  1. Are you inside Ubuntu? If your prompt mentions PS C:\, you're in PowerShell — type wsl first. Your key lives in Ubuntu, not Windows.
  2. Did the key actually get added? Open github.com/settings/keys. If the list is empty, redo Stage 2 steps 2–3. A common miss: copying but never clicking Add SSH key.
  3. Did you paste the whole key? It must start with ssh-ed25519 and end with your email. Re-copy it: 
    cat ~/.ssh/id_ed25519.pub
    Delete the old key on GitHub and add this one again.
  4. Cloned with the wrong URL? Run git remote -v inside the project. If it shows https://github.com/..., switch it to SSH: 
    git remote set-url origin git@github.com:simonteague6/profile-card.git

Then test again with ssh -T git@github.com.

😬 "I committed to main by accident"

Very fixable, and nothing is lost. You made your commit on main instead of a branch. This recipe moves your work to a proper branch and puts main back the way it was. Run the four commands one at a time:

git switch -c feature/add-your-username-card

Creates your branch right where you are — your commit comes with it. Your work is now safe.

git switch main
git reset --hard origin/main

Rewinds your local main to match GitHub's — as if the accident never happened. (Careful: this throws away any uncommitted edits on main, so save/commit anything you care about first.)

git switch feature/add-your-username-card

Back on your branch, with your commit intact. Continue from Stage 7.

⚔️ Merge conflict — "CONFLICT" or GitHub says the branch can't be merged

A conflict means you and someone else changed the same lines of the same file, and git needs a human to pick. Since everyone adds their own file in cards/, this is rare here — but if it happens:

  1. On your branch, pull in the latest main: 
    git fetch origin && git merge origin/main
  2. Git tells you which files conflict. Open them in Zed — you'll see markers like this: 
    <<<<<<< HEAD
your version of the line
=======
their version of the line
>>>>>>> origin/main
  3. Edit the file so it reads how it should — keep yours, keep theirs, or blend them — and delete all the <<<<<<<, =======, >>>>>>> lines. Zed highlights conflicts and offers one-click "keep" buttons.
  4. Then seal it and push: 
    git add . && git commit -m "fix: resolve merge conflict" && git push
🚫 Push rejected

Read the error message — it's one of two situations:

"Permission to simonteague6/profile-card.git denied" or a 403 error → you're not a collaborator on the repo yet. Check github.com/notifications and your email for the invite (Stage 3, step 1), accept it, and push again. No invite anywhere? Send Simon your GitHub username.

"[rejected] ... failed to push some refs" with a hint about "fetch first" → GitHub has commits on your branch that your laptop doesn't (this can happen if you edited a file on the GitHub website, or pushed from another machine). Download them, then push again:

git pull --no-rebase && git push

If the pull triggers a conflict, the merge-conflict entry above takes over. If the output is scary and nothing works, screenshot it and ask — this one's worth a second pair of eyes.

🐧 Commands say "not recognized" or nothing from this page works Windows

You're almost certainly in PowerShell, not Ubuntu. Type wsl, press Enter, then cd ~. Every command on this page assumes you're inside Ubuntu. (Zed's built-in terminal is already inside Ubuntu when you open the project via WSL Remote.)

🔍 "zed: command not found" macOS

The zed terminal command isn't installed yet (common if you downloaded Zed directly instead of using Homebrew). Open Zed, press Cmd+Shift+P, type install cli, and run it. Works immediately — no restart needed.
